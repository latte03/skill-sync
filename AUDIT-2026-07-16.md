# SkillSync 当前实现审计报告

> 审计日期：2026-07-16  
> 审计依据：当前 `main` 分支的代码、测试和公开设计文档。  
> 未使用或引用历史 `ANALYSIS.md` 的任何结论。

## 结论摘要

项目的核心方向是合理的：以本地文件系统为持久化基础、以 Core 层服务 CLI 和 HTTP API、以 `SkillKey` 解决多来源同名冲突。近期补上的原子写入、状态锁、分发/删除事务日志，已显著降低分发路径的中断风险。

产品下一阶段不应继续泛化地扩展恢复机制。依赖执行安全和 Dashboard 缺失设置 API 两个 P0 已完成；现在应进入 **更新 API 与用户可见功能**，同时把 `switch` 的原子恢复留在独立 P1 队列中。

## 实施状态（本次审计后的第一批改动）

- P0-1 已处理（`443a87a`）：包依赖默认只提示不执行；显式 `--install-deps` 才会运行，且改为参数数组并禁用 npm lifecycle scripts。
- P0-2 已处理（`443a87a`）：Git 平台和网络代理 API 已补齐，平台选择互斥，平台凭证改写入 `secrets.yaml`。
- 未完成的 P1（版本恢复事务、本地导入替换、读请求恢复、doctor 重建、完整 API 等价）仍保留为后续独立工作项。

### 当前真实模型

```text
SkillKey
  └─ skills/<skill-key>/
       ├─ SKILL.md 与 skill 文件
       ├─ manifest.yaml        （skill 级元数据）
       └─ .backup/             （版本备份）

skills-lock.json               （全局索引、来源快照、分发状态）
tags.yaml                      （标签倒排索引）
Agent skills 目录              （symlink/junction/copy 投影）
```

- `SkillKey` 是唯一且不透明的标识；GitHub Skill 使用 `owner/repo/skill-path`，本地 Skill 使用规范化相对路径。
- `namespace` 已不在运行时代码、数据模型或 HTTP API 中使用；仅旧的一次性迁移脚本仍会写入 `namespace: local`。
- 分发、取消分发、单目标链接替换和删除中央 Skill 已使用可恢复的目录替换与日志。

## 问题与建议

### P0-1（已完成）：第三方 Skill 的依赖声明可触发非预期命令执行

**证据**

- `src/core/installer.ts` 仅在显式 `installDeps` 时调用 `installDependencies`。
- `src/lib/dependencies.ts` 使用 `execFileSync` 的参数数组；npm 安装固定使用 `--ignore-scripts`。
- `tests/lib/dependencies.test.ts` 覆盖了参数数组和 lifecycle script 禁用行为。

**影响**

修复前，远端 frontmatter 会成为命令参数来源，且包安装无需明确授权。

**处理建议**

已改为默认跳过、`--install-deps` 显式授权、参数数组执行、npm lifecycle scripts 禁用；显式安装失败会使安装失败。

### P0-2（已完成）：Web 客户端调用的 Git 平台与代理 API 在服务端不存在

**证据**

- `src/server/routes/settings.ts` 已实现 `/api/git/platforms*` 和 `/api/network/proxy`。
- `SyncView.vue`、`SettingsView.vue` 的既有调用已与这些路由对齐。
- `tests/server/app.test.ts` 覆盖平台互斥、凭证保存、代理校验和无效平台请求。

**影响**

修复前，同步页的 Git 平台设置、设置页的 Git 凭证和网络代理会直接收到 404。

**处理建议**

已作为第一批 API 交付；同时将 GitHub/Gitee 凭证迁移至 `secrets.yaml`，并将平台启用状态改为互斥。

### P1-1：`switch` / 从备份恢复仍不是可崩溃恢复的事务

**证据**

- `src/core/version-manager.ts:301-328` 先逐项删除中央目录内容，再复制备份，最后分别写 manifest 和 lock。
- `src/core/state-recovery.ts:12-20` 仅恢复分发和删除日志，不覆盖版本恢复。

**影响**

在目录删除、备份复制、manifest 写入或 lock 写入之间中断，中央 Skill 可以残缺，元数据也可能与文件不一致。

**处理建议**

这项应作为独立的 P1 技术债：采用临时目录完整复制 + 目录替换，并复用统一的恢复入口。它不应再阻塞 API 功能开发。

### P1-2：本地安装和导入会合并已有目录，而非替换完整快照

**证据**

- `src/core/installer.ts:423-427` 对本地安装使用 `mkdir` + 递归复制。
- `src/core/skill-manager.ts` 的导入路径同样先复制到目标目录，再写 manifest/lock。
- GitHub 安装已经使用 staging 目录和 `replaceSkillSnapshot`，两条路径的语义不一致。

**影响**

重装同一 Skill 时，来源中已经删除的旧文件会残留；导入同 key 还可能覆盖 metadata 而没有明确的 replace/冲突决策。

**处理建议**

把本地 install/import 也统一为“发现 → staging → 原子替换 → metadata 提交”，并要求调用方明确 `replace` 策略。

### P1-3：恢复只在 CLI 全部命令与 HTTP 写请求前运行，HTTP 读请求可能观察到中断状态

**证据**

- CLI 在 `src/bin/skill-sync.ts` 的 `preAction` 中恢复状态。
- `src/server/app.ts:71-86` 的恢复中间件只匹配 POST/PUT/PATCH/DELETE。
- Dashboard 的 `/api/status`、`/api/skills` 等 GET 请求可在崩溃后先于任何写操作执行。

**影响**

Web Dashboard 首次打开可能读到“目录已移动、metadata 尚未提交”或“metadata 已提交、旧目录未清理”的过渡状态。

**处理建议**

在服务启动时恢复一次，或让所有 `/api/*` 请求在只读、幂等的恢复检查后再继续；同时保留状态锁冲突的 409 语义。

### P1-4：声明的“manifest 为真相源、lock 自动修正”没有实现闭环

**证据**

- PRD 和 `AGENT.md` 都声明 manifest 是 skill 级真相源、与 lock 冲突时自动修正。
- `src/core/skill-manager.ts` 的查询以 `skills-lock.json` 枚举 Skill。
- `src/commands/doctor.ts:180-197` 只检查“lock 指向的目录是否缺失”，不检查 manifest/lock 字段一致性，也不提供重建。

**影响**

lock 丢失、manifest 存在但未被索引、来源/版本/分发字段漂移时，系统没有宣称的自动恢复能力。

**处理建议**

明确并实现一个 `doctor --repair`：扫描 `skills/**/manifest.yaml`、校验 SkillKey、生成差异报告；修复必须要求明确确认。若不计划实现，应从文档删除“自动修正”的承诺。

### P1-5：API 契约仍是“薄封装”的部分实现，行为与 CLI 不等价

**证据**

- API 分发在 `src/server/app.ts:321-341` 固定 `mode: 'symlink'`，未暴露 copy/force/dry-run。
- 删除 API 在 `:670-687` 只接受 `all`/`central`，无法表达 CLI 的 `remove --agent`。
- API 没有 update、switch/restore、备份列表的写操作端点。
- 错误响应有 `apiError` 与手写 `{ error }` 两种风格，部分状态锁错误会退化为 500。

**影响**

API 不能稳定地成为 CLI 的正式功能层，新功能只能继续绕过 API 或在前端堆分支。

**处理建议**

定义 `/api/v1` 契约：请求/响应 schema、统一错误信封、操作参数和异步任务结果。先覆盖当前 Web 已依赖的路径，再按 CLI 能力扩展。

### P2-1：文档和实现有可修正的过期差异

| 文档承诺 | 当前实际 | 建议 |
|---|---|---|
| `--deploy <type>` | CLI 实际提供 `--copy`，无 `--deploy` 选项。 | 选择一个接口并统一 PRD、命令注释和帮助文本。 |

| namespace | 正式实现已删除；旧 `scripts/migrate-skills.mjs` 仍会写 `namespace: local`。 | 修正或归档该一次性脚本。 |

### P2-2：`clean --orphans` 对嵌套 SkillKey 的扫描仍只有顶层视角

**证据**

- `src/commands/clean.ts:119-141` 仅枚举 Agent skills 根目录的第一层。
- SkillKey 可包含多层路径；当前实现只能保护已知顶层前缀，不能准确发现或清理深层孤儿。

**处理建议**

若保留嵌套 SkillKey，把 orphan 识别改为递归遍历，并且永远在删除前展示完整相对路径和 dry-run 结果。

## 已完成：来源关联与单 Skill 更新 API

本地 Skill 不应永久被判定为“不可更新”。下一批把它升级为“**可发现候选远端来源，但必须由用户确认关联**”，然后才进入更新闭环。绝不根据名称自动绑定，更不在关联阶段覆盖本地文件。

已交付：

1. 新增 `routes/updates.ts`，提供单 Skill 的更新检查、更新执行和备份列表 API；所有 SkillKey 继续使用 query 参数，不拆分路径。
2. 为本地 Skill 提供候选发现 API：以 frontmatter `name`、描述和当前 SkillKey 生成查询，调用既有 skills.sh 搜索；候选仅用于展示和排序，不产生写入。
3. 为候选提供远端验证：将 skills.sh 的 `source`/`skillId` 解析为 GitHub owner、repo、skillPath；只有能从 GitHub Trees API 确认存在 `SKILL.md` 的候选，才允许提交关联。搜索结果无法唯一定位 repo 时，允许用户提供 GitHub URL 作为候选来源。
4. 新增显式关联 API：用户选择一个已验证候选后，仅更新 manifest/lock 的来源字段为 GitHub 位置，保留当前本地目录和版本；将 `treeSha` 清空，使首次 `check` 明确显示可更新。关联操作必须具备 manifest/lock 回滚保护。
5. 更新写操作暴露 `force` 与 `noBackup`，统一复用 `apiError` 的 400/409/500 契约。关联后的更新仍走既有备份和更新流程。
6. 增加 API/Core 测试：无候选、多个候选、远端验证失败、确认关联、首次检查可更新，以及 API 参数校验。

明确不在本批范围：

- 不暴露 `switch`/恢复 API，直到 P1-1 的目录替换与崩溃恢复完成。
- 不做批量更新 API；现有 CLI 的逐项语义保持不变，等单 Skill 更新闭环验证后再扩展。
- 不做泛化任务队列；只有出现真实的长时间、多步骤操作需求时再引入持久化任务模型。
- 不修改现有 Vue 页面；本批只稳定 Core 与 API，不执行 `pnpm build:web`。

### 关联判定规则

| 情况 | 系统行为 |
|---|---|
| 没有远端候选 | 维持 `local`；不纳入更新范围。 |
| 一个或多个候选 | 展示来源、描述、stars、installs 和验证后的 GitHub 路径；用户选择后才关联。 |
| 本地内容与远端不一致 | 仍可关联，但必须明确告知“关联不会覆盖本地内容”；首次 update 由现有备份/force 规则决定。 |
| 无法验证候选 GitHub 路径 | 不允许关联，保留为搜索结果。 |

### Web 完全重构的边界

Web 重构安排在上述 API 稳定之后，作为独立产品阶段，而不是继续在现有页面上增量修补。届时先确定新的信息架构、状态模型和 API client 边界，再重建应用壳、技能详情、更新/关联流程和设置页；旧页面只作为行为参考，不作为组件改造基础。

## 已完成：API 契约补齐与显式依赖审阅

- 分发 API 已支持 `mode`、`force`、`dryRun`，不再固定为 symlink；取消分发也支持 `dryRun`。
- 删除 API 已支持 `scope=agent&agent=...`，与 CLI 的单 Agent 删除语义对齐。
- 已补齐 `/api/skill/dependencies` 只读审阅和 `/api/skill/dependencies/install` 显式安装接口；后者只接受已持久化声明中的 `npm`/`pip` 管理器选择，不能由客户端传入包名。
- 依赖安装仍保留已有安全约束：默认不执行、必须显式请求，npm lifecycle script 禁用。

API 尚未提供 `switch`/恢复端点，这是因为 P1-1 的崩溃恢复尚未完成。下一阶段进入完整 Web 重构；其后处理 `switch` 原子恢复。

## 建议的近期验收标准

- 远程 Skill 的依赖不会在未经批准时执行命令或 lifecycle script。
- Web 中 Git 平台和代理设置不再出现 404，并有服务端测试覆盖。
- 每个写 API 使用同一错误信封；状态锁稳定返回 409。
- API 可以明确表达 SkillKey、目标 Agent、分发模式、强制覆盖和删除范围。
- 继续保持 `pnpm test` 与 `pnpm build` 为核心验证；未修改 Web 时不运行 `pnpm build:web`。
